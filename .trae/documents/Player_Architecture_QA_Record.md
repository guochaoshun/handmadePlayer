# 手搓 iOS 播放器与防盗链底层架构探索记录

> 本文档记录了从零手写 iOS 播放器，并实现基于 FFmpeg AVIOContext 拦截的防盗链解密方案的全过程技术探讨。涵盖了音视频底层 I/O、YUV/RGB 渲染原理、音视频同步及多线程架构等硬核知识点。

## 1. 播放器手搓初探与 FFmpeg 的价值

**答：**
这是一个非常有挑战性但也极其锻炼底层技术的想法！音视频开发是客户端开发中门槛较高、但也最硬核的领域之一。

我们先从 **ijkplayer** 聊起，然后逐步拆解如果你要在 iOS 上“手搓”一个播放器，需要跨越哪些山海。

### 一、ijkplayer 简介与核心价值
**ijkplayer** 是 Bilibili 开源的一款基于 FFmpeg 的跨平台（iOS/Android）轻量级视频播放器。它在移动端音视频领域有着“祖师爷”级别的地位。
- **核心架构**：它的底层核心逻辑其实是对 FFmpeg 自带的测试播放器 `ffplay` 的移动端移植与重构。`ffplay` 解决了极其复杂的音视频同步问题，ijkplayer 在此基础上封装了移动端的上层接口。
- **最大贡献**：
  - **跨平台一致性**：一套 C 语言底层的核心播放逻辑，跑在 iOS 和 Android 上，保证了双端表现的高度一致。
  - **接入硬件加速**：在 iOS 上接入了 `VideoToolbox`，在 Android 上接入了 `MediaCodec`，弥补了原生 FFmpeg 在移动端硬解支持较弱的问题。
  - **渲染层改造**：针对移动端重写了渲染层（iOS 上使用了 OpenGL ES，不过现在更推荐 Metal）。
- **现状**：目前 B 站官方基本已经停止了大规模维护，但它依然是无数中小型公司自研播放器的基石，也是学习播放器开发的最佳开源教材。

### 二、手搓 iOS 播放器：核心链路与知识储备
一个完整的播放器，本质上是一个**流水线（Pipeline）**工程。数据从网络或本地进来，最终变成屏幕上的画面和喇叭里的声音，经历以下四个核心阶段：
1. **解协议与解复用（Demuxing）**：把 MP4、FLV、m3u8 等“容器”，剥离出单纯的视频流和音频流。
2. **解码（Decoding）**：把压缩的音视频流，还原成原始数据（视频变成 YUV/RGB，音频变成 PCM）。
3. **音视频同步（A/V Sync）**：决定哪一帧画面应该在什么确切的微秒显示，哪一段音频应该在什么时刻播放。
4. **渲染与播放（Rendering & Playback）**：把 YUV 画面画到屏幕上，把 PCM 数据塞给扬声器。

### 三、你可能遇到的最大困难（“三座大山”）
1. **音视频同步（最难的 Boss）**：需要深刻理解 PTS 和 DTS，建立主时钟（通常是“视频同步到音频”）。
2. **内存管理与性能调优**：音视频数据一旦解压，体积是惊人的。需要自己设计一套线程安全的帧缓冲队列。
3. **状态机与边界情况处理**：如 Seek（拖动进度条）、前后台切换、弱网缓冲等。

## 2. 深入探讨：为什么大家都离不开 FFmpeg

**问：**
针对这个事情,我可不可以说。所有的播放器。所有的开源播放器 底层都是采用FFmpeg实现。

**答：**
这个说法**不能说 100% 准确，但可以说覆盖了“跨平台”和“全格式”领域的 90%**。

世界上存在**两大阵营**：

#### 第一大阵营：操作系统“亲儿子”阵营（坚决不用 FFmpeg）
苹果、谷歌、微软的原生播放器，底层绝对没有 FFmpeg 的一行代码。
*   **iOS的 `AVPlayer`**：底层是苹果自己的 `CoreMedia` 和 `VideoToolbox`。
*   **特点**：极其省电、流畅，但**极其挑食**。不支持 MKV、FLV、RTMP 等格式。

#### 第二大阵营：“民间野生”与“跨平台全能王”阵营（全部是 FFmpeg 的信徒）
只要号称“什么格式都能播”、“跨平台”，底层核心几乎 100% 是 FFmpeg。
*   **ijkplayer / VLC Player / mpv**
*   **各大直播平台的自研播放器**
*   **Chrome 浏览器的 media 模块**

**为什么你的加密需求必须选 FFmpeg？**
如果你用苹果的 `AVPlayer`，它闭源且不提供“在读取文件时拦截并解密”的高效底层 I/O 接口。而 FFmpeg 提供了 `AVIOContext`，你可以极其优雅地写几十行 C 代码，直接在最底层插入 AES 解密逻辑，上层什么都不用改。

## 3. 防盗链实战：文件系统与指针回溯的陷阱

*(在讨论完 AVIOContext 的 `read_packet` 拦截后，探讨了指针和缓冲区的细节)*

**问：**
`int64_t logicalOffset = lseek(ctx->fd, 0, SEEK_CUR) - ctx->dirtyLength;`
`int bytesRead = (int)read(ctx->fd, buf, buf_size);`
这一步从物理文件读取数据的过程中,为什么没有使用逻辑地址。

**答：**
这是一个非常敏锐的问题！直接抓住了底层 I/O 拦截机制的核心：**文件指针的状态管理**。
操作系统的文件描述符（`fd`）本身是带有“状态”的，内部维护了一个“当前读写指针”。
`read(fd, buffer, size)` 会从当前指针位置开始读，读完后自动向后移动。你不需要每次都告诉它从哪里读。
保证 `fd` 物理指针永远正确的，是我们初始化时强行越过脏数据的 `lseek`，以及在 `seek_packet` 中做好的精确映射。计算 `logicalOffset` 仅仅是为了做业务判断（是否落在加密区）。

**问：**
`/Users/uxin/Desktop/无名高地/fileReadAction/GCSPlayer/GCSPlayer/YCCorePlayer.m#L37-51` 我这样改（把 lseek 放到 read 之后）, 是不是逻辑完全不受影响, 依然正确.

**答：**
**不，这样改逻辑就错了！这会引发极其隐蔽且致命的 Bug！**
如果在 `read()` 执行之后再去获取当前物理偏移量，拿到的是**读取后**的新位置，而非本次读取的**起始**位置。如果基于这个错误的位置去判断是否落在加密区，解密的数据段会完全错位。
必须在 `read` 发生之前，先把当前的“起始位置”记录下来（事前拍照法）。

**问：**
或者说我可以用另一个思路。计算逻辑地址的时候再额外减一个buf_size, 这样是不是也可行

**答：**
方向是对的，是典型的“事后回溯法”。但**绝对不能减 `buf_size`！必须减 `bytesRead`！**
在 POSIX 规范中，`buf_size` 是你想读多少，而 `bytesRead` 是实际读到了多少。如果遇到文件末尾（EOF）或者被信号中断，发生了**短读（Short Read）**，`bytesRead` 会小于 `buf_size`。如果减去 `buf_size` 会算出负数或错误的地址。

lseek(ctx->fd, 0, SEEK_CUR)这个函数的作用。入餐和返回职也写到注释里。方便我后续理解。

**答：**
*(已在代码中补充了极其详尽的 Hacker 用法注释。短读是 POSIX 标准常态，利用 FFmpeg 框架的健壮性，遇到跨界读取时直接截断返回，让 FFmpeg 自动发起下一次读取，是代码最精简、最安全的做法。)*

## 4. 防盗链物理结构确认

**问：**
有一个问题我需要确认一下。咱们在头部写入了30字节的脏数据。同时又对头部的1 kb数据进行了加密。那么这三十个字节在这个加密的范围内吗?还是脏数据独立于加密的范围。

**答：**
**这 30 个字节是独立于加密范围的！它们不属于那 1KB 的加密数据。**
物理文件结构图如下：
```text
物理文件起始点 (Offset: 0)
|--- 第 1 段：脏数据区 (Dirty Data) [30字节，明文噪音]
物理偏移量 (Offset: 30) = 逻辑偏移量 (Offset: 0)
|--- 第 2 段：加密区 (Encrypted Header) [1024字节，AES密文]
物理偏移量 (Offset: 1054) = 逻辑偏移量 (Offset: 1024)
|--- 第 3 段：明文区 (Plaintext Body) [剩余所有字节，明文音视频帧]
```
这种分层防御（格式破坏 + 局部加密）性价比最高，且脏数据不参与 AES 运算，避免了 block 对齐的复杂性。

## 5. 解码工作台与 I/O 接管

**问：**
`avio_alloc_context` 这个函数的作用是什么?

**答：**
这是整个“透明解密”能够成立的**灵魂枢纽**。它的作用是**把底层读取数据的控制权，从 FFmpeg 手里“抢”过来，交给你自己写的函数。**
通过将它赋值给 `_formatContext->pb`（协议缓冲区），FFmpeg 就不再自己去调系统底层的 `open` 读文件，而是乖乖地去调用你塞给它的 `read_packet` 函数。

**问：**
那我理解一下,如果想要做成一个网络播放器的话,也是需要重写这个read_packet方法。在这个方法里发起网络请求。带有range参数。然后接二进制流。再返还给播放器。大致是这样的流程吗?

**答：**
大方向完全正确！但有一个生死坑：**绝对不能在 `read_packet` 里直接同步发起网络请求。**
因为 `read_packet` 是在 FFmpeg 的解析线程里同步执行的，会被网络 I/O 严重阻塞导致播放器卡死。
工业级的做法是：引入**环形缓冲区（Ring Buffer）**。起一个独立的下载线程拉数据塞进缓冲区，`read_packet` 只负责极速从内存缓冲区“捞”数据，没数据就挂起等待。

**问：**
`_formatContext->nb_streams` 这部分完全看不明白要干什么。顺带解释一下setupCodecContext这个函数的作用。

**答：**
MP4 文件就像多车道高速公路。遍历 `nb_streams` 就是为了从混杂的车道中，挑出我们需要的**视频车道**和**音频车道**，并记住它们的 Index，以便后续分发 Packet。
`setupCodecContext` 则是为找出的流搭建专属的**解码工作台 (`AVCodecContext`)**。它负责：找对解码师傅（AVCodec） -> 建工作台 -> 抄写图纸参数 -> 正式开工。

**问：**
`avformat_find_stream_info` 为什么要探测这个信息?不探测会怎么样。

**答：**
不探测大概率会解码失败。`open_input` 只是看了个容器封面，不知道视频真实宽高、FPS、时间基，更缺少解码极其依赖的 `extradata` (如 H.264 的 SPS/PPS)。`find_stream_info` 会在底层悄悄试读和试解码，把这些决定生死的关键信息探测出来。

## 6. YUV 与 RGB 的底层哲学

**问：**
为什么需要这个转换。为什么编码的时候是yuv。而不能直接编码成rgb。这样不就不用做转换了吗?如果可以的话,再介绍一下yuv和rgb的历史。假如说彻底抛开历史包袱,让你重新设计一套编码,你还会使用yuv吗?为什么屏幕硬件指认rgb。他为什么不能改成认yuv呢?

**答：**
即使抛开兼容黑白电视的历史包袱，现代视频编码依然会毫不犹豫地选择 YUV（或类似空间）。
1. **为什么编码必须用 YUV？**
   * **数学去相关性**：RGB 三通道高度相关，存在极大信息冗余。YUV 转换是一种能量集中变换，把 90% 能量集中在 Y（亮度），使得 UV（色差）能量极低，极其适合极限压缩。
   * **生理学欺骗**：人眼对亮度敏感，对色彩迟钝。YUV420 采样直接丢弃 75% 的色度信息，数据量物理砍半，人眼却看不出受损。RGB 做不到这种有损压缩。
   * **优雅的可伸缩性**：网速差时可以牺牲色彩（UV）保全轮廓（Y）。
2. **为什么屏幕只能认 RGB？**
   * **物理发光限制**：屏幕是发光设备，光的三原色是红绿蓝（加法混色）。物理世界造不出能发射“蓝色差光（相对值/负数）”的灯管。
   * **生物学解剖限制**：人类视网膜上有三种视锥细胞，分别对红绿蓝光最敏感。屏幕只需发射这三种光就能完美欺骗大脑。
3. **软件与硬件的伟大分工**：
   传输存储在“软件世界”用 YUV 极限压缩；最终显示在“物理世界”必须用 RGB 刺激人眼。GPU 的 Fragment Shader 就是连接两者的桥梁，瞬间完成翻译。视频的美颜、滤镜，也全都是在这个 Shader 桥梁的矩阵变换中实现的。
